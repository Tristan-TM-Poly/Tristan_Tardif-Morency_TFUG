# Native First Doctrine

## Core principle
Use what is native to us whenever the gain is real, measurable, and stable.
Keep Python as a thin orchestration shell whenever a lower-level language can own the hot path better.

## Language allocation
- Rust: safe native core, structured data pipelines, synergy engines, packet logic, bounded inference kernels
- C++: numerical kernels, spectral LDK kernels, sparse graph/hypergraph math, tight memory control
- Go: service daemons, concurrent orchestration services, long-running workers
- TypeScript/Tauri: operator cockpit, dashboards, mission control, human-facing control planes
- Python: wrappers, validation, experiments, migration adapters, tests

## Migration law
Do not rewrite everything at once.
Migrate the hottest and most sovereignty-critical paths first.

## Selection criteria
Promote a path to native when it has:
- repeated heavy loops
- combinatorial growth
- string-heavy data that should be interned
- expensive dynamic dispatch
- strategic reuse potential across AI-7, TRISTAN2, and TFUGA

## Governance law
Every native migration must keep:
- packet parity
- score parity
- review parity
- rollback path
- benchmark evidence

## Immediate native-first targets
- synergy engines
- bounded inference kernels
- dataset-aware ranking
- pilot scoring pipelines
- LDK spectral kernels
